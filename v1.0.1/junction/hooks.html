junction.hooks – Extension Points for Hub Functionality¶

junction.hooks.connection_lost(peer, subscriptions)¶

A connection has gone down unexpectedly

Parameters:
  • peer ((host, port) tuple) – the (host, port) with which the peer identified itself
  • subscriptions – the subscriptions the peer had when it went down; this is a list of four-tuples of (msg_type, service, mask, value) where msg_type may be 4 for publish, or 5 for rpc (these constants are found in junction.core.const).
junction.hooks.connection_received(peer, subscriptions)¶

A connection was received from a Hub that was not in our to_connect list

Parameters:
  • peer ((host, port) tuple) – the (host, port) with which the peer identified itself
  • subscriptions – the subscriptions the peer sent in its handshake message; this is a list of four-tuples of (msg_type, service, mask, value) where msg_type may be 4 for publish, or 5 for rpc
junction.hooks.select_peer(peer_addrs, service, routing_id, method)¶

Choose a target from the available peers for a singular message

Parameters:
  • peer_addrs (list) – the (host, port)``s of the peers eligible to handle the RPC, and possibly a ``None entry if this hub can handle it locally
  • service (anything hash-able) – the service of the message
  • routing_id (int) – the routing_id of the message
  • method (string) – the message method name
Returns:

one of the provided peer_addrs

There is no reason to call this method directly, but it may be useful to override it in a Hub subclass.

This default implementation uses None if it is available (prefer local handling), then falls back to a random selection.
